## 1. 目标(一句话)

一个**最小、响应式**的 React 控制面 UI:登录、看近期运行+状态、看某次简报、管理排程(列表/新增/启停/改/删)、立即触发一次运行。通过 HTTP 调用 Unit 1 的 API。

## 2. 两个关键设计点(计划里务必说清楚)

**① 会话 + CSRF 的前端消费(对上 Unit 1 的契约)**
- 所有请求带 `credentials: "include"`(带上会话 cookie)。
- **写操作(POST/PATCH/DELETE)**:从 JS 可读的 `csrftoken` cookie 取出 token,放进 `X-CSRF-Token` 请求头。
- 统一处理 **401**(跳回登录)和 **403**(CSRF/权限)。
- 这是和 Unit 1 对接的命门——**说清楚你怎么读 cookie、怎么带头、怎么集中处理这两个状态码。**

**② 异步运行的状态反映**
- 点"立即跑" → `POST /runs` 返回 **202 + 一条 pending Run**;真正执行在 worker 进程。
- 所以 UI **不能假设点完就有结果**:要**轮询** runs 列表/详情,把状态从 `pending → running → success/failed` 反映出来,跑完能看到产出。
- **说清楚轮询策略**(间隔多少、什么条件下停止轮询),别写成永不停的死循环。

## 3. 范围(MVP)

**做(In):** 登录页;运行看板(近期 runs + 状态);运行详情 + 看产出(digest 渲染);排程管理(列表 + 新增 + 启停 + 改 cron/tz + 删除);"立即跑"按钮;**响应式**(手机能看)。

**不做(Out):** ❌ 改后端 ❌ 上云(后续独立步) ❌ 多用户 ❌ 多 agent/meta-agent ❌ 花哨图表/手机专属界面。

## 4. 技术栈与设计要求(默认,计划里可微调)

**技术栈:**
- **Vite + React**(Unit 1 的 CORS 默认就指向 Vite 的 `http://localhost:5173`)。
- 样式从简,自己写 CSS 或用轻量方案即可;**别堆重型组件库**。
- 数据获取/状态:**保持简单**(React state + `fetch`;想用 React Query 也行,别过度工程)。

**设计(自用控制面板:干净好用,不求惊艳):**
- 干净的**管理面板**观感:清晰的视觉层级,留白和字体舒服(别挤),整体克制、不花哨。
- **运行状态**明显可辨:`pending / running / success / failed` 各自的颜色或徽章,一眼能区分。
- **响应式从一开始就做**:手机竖排单列(以"看"为主),桌面表格/网格(以"操作"为主)。
- **digest 产出**渲染得好读(markdown → 排版,而不是一坨纯文本)。
- 状态齐全:**loading / 空列表 / 错误**(401 跳登录、请求失败给提示),别只画 happy path。
- 组件样式保持一致(按钮/表单/卡片统一风格)。

## 5. 约束

- 前端是**独立 app**(放 `frontend/` 子目录),只通过 HTTP 调 Unit 1 的 API;**不碰后端代码**。
- 后端 **111 个测试保持绿**(前端不动后端)。
- 配置(API base URL 等)走前端 env(Vite `VITE_` 前缀),不硬编码。
- **本地跑通即可**(Vite dev server + 本地 API);上云是后续独立一步。

## 6. 测试

- 关键路径的**基本测试**(登录、列表渲染、写操作带 CSRF 头),用 Vitest + React Testing Library 之类;**别过度投入**。
- 后端测试不受影响。

## 7. 验收标准

1. 能登录(错密码被拒、未登录看不到内容);登出可用。
2. 看板列出真实 runs + 状态;能打开某次运行看 digest。
3. 排程:能新增/启停/改/删,变更落到 DB(正在跑的 scheduler 能动态捞到)。
4. **"立即跑" → 看板出现 pending 的 run → worker 跑完后 UI 轮询到状态变 success + 能看产出**(端到端走通 Unit 1 的交接)。
5. 手机宽度下可用。
6. 后端未动、111 测试仍绿。
