# 个人 Agent 系统 · 架构与建造蓝图

> **这份文档是整个系统的"单一真相来源 / 北极星"。**
> 用途有两个:(1) 描述系统的完整目标架构;(2) 作为你发给每个 build session 的总图,让它们对齐同一套设计和契约。
> **你(人)是 orchestrator**;这份文档 + 里面定义的契约,就是协调多个 build session 的"胶水"。
> 原则:**北极星 = 完整愿景,给方向;但实现一寸寸来(见第 8 节路线图)。**

---

## 1. 系统目标与范围(阶段 0)

**一句话**:一个**云端托管、登录即用的控制台**,用来运行、排程、监控和管理我的个人多 agent 工作流系统。

**完整能力(愿景)**:任何设备打开网页、账号密码登录,就能——
- 运行 / 排程工作流
- 查看任务状态与产出
- 安排新任务、处理系统相关事务
- 监控运行情况
- 设备自适应:**电脑 / 平板 → 编排工作流(重操作);手机 → 监控等轻操作**

**首个落地工作流(MVP 锚点)**:每天 6:00 抓取指定来源新闻 → 解析 → 推送给我。

**学习目标**:借这个项目练 agent 工程(记忆、权限、工作区、编排)+ 走一遍软件工程四阶段。

**明确延后 / 非目标**:
- meta-agent(建 agent 的 agent)——最后做
- 浏览器 / 社媒 agent——后期,且需先过平台 ToS 合规
- 重度多 agent 并行——后期

## 2. 北极星架构(阶段 2)

系统劈成三个面,**核心是"控制面"与"执行面"解耦,中间靠数据面传话**。

```
          ┌─────────────────────────────────────────────┐
  你 ───▶ │  控制面 Control Plane(你登录的网页)         │
 任意设备 │   React 前端  ──REST/OpenAPI──▶  FastAPI 后端 │
          │   (认证后:看状态/排任务/看产出/监控)         │
          └───────────────┬─────────────────────────────┘
                          │  读写"定义/排程/状态/产出"
                          ▼
          ┌─────────────────────────────────────────────┐
          │  数据面 Data Plane = 单一真相来源             │
          │   PostgreSQL(+ 可选任务队列)                │
          └───────────────┬─────────────────────────────┘
                          │  worker 领取待执行的 run、写回状态/产出
                          ▼
          ┌─────────────────────────────────────────────┐
          │  执行面 Worker Plane(真正干活的 agents)     │
          │   Agent SDK worker  ◀── Scheduler(定时触发)│
          │   ├─ 调外部:新闻源、通知渠道(邮件/TG)      │
          │   └─ 后期:浏览器/社媒、子 agent 编排         │
          └─────────────────────────────────────────────┘
```

**关键解耦原则**:网页**不直接跑 agent**;网页把"任务定义 / 排程"写进 DB → worker 从 DB 领取并执行 → 把状态和产出写回 DB → 网页再读出来显示。

**为什么必须这样**:
- 工作流是**长时间、定时**的,塞不进一个网页请求;
- 网页要**随时响应**(手机点一下立刻有反应);
- agent 要能**没人看着也继续跑**(出门在外、网页关着,6 点照跑)。

**这条解耦就是系统的脊椎**,也是控制面与执行面之间的契约(DB/队列 = 共享真相)。第一天就定死。

## 3. 核心设计:定义即"一等数据"

**把"agent 定义""工作流定义""排程"做成系统里的一等数据(存 DB),不要写死在代码里。**

- 现在:你通过 UI(或手动)创建这些定义。
- 以后:meta-agent 不过是**同一份定义数据的另一个写入者**——它生成新 agent/工作流,就是往同一张表里写。

这是为未来 meta-agent **现在就免费留好的缝**:声明式定义 → 谁创建都行(人 / UI / meta-agent)。

---

## 4. 数据模型草案(DB 契约 · 概念层,非建表语句)

凡是碰数据的模块,都以这套实体为契约:

| 实体 | 作用 | 关键字段(概念) |
|---|---|---|
| **User** | 认证 | id、账号、密码哈希、角色 |
| **AgentDef** | agent 定义(声明式) | id、名称、系统提示、允许的工具、权限策略、**provider/model(Claude / 非 Claude)** |
| **WorkflowDef** | 工作流定义(声明式) | id、名称、步骤序列、每步用哪个 AgentDef、参数 |
| **Schedule** | 排程 | id、关联 WorkflowDef、cron 表达式、是否启用 |
| **Run**(Task) | 一次执行 | id、关联 WorkflowDef、状态(待执行/进行中/成功/失败)、起止时间、触发方式 |
| **Output**(Artifact) | 产出 | id、关联 Run、类型、内容/存储位置 |
| **Event**(Log) | 审计 / 监控 | id、关联 Run、时间、级别、消息 |
| **Memory** | 跨 run 的长期记忆 | id、关联 agent/主题、内容(后期加 pgvector 向量字段) |

关系要点:WorkflowDef 引用多个 AgentDef;Schedule 指向 WorkflowDef;每次触发产生一个 Run;Run 产生 Output 和 Event;Memory 跨 Run 持久。

## 5. 关键契约 / 边界(并行开工前必须定死)

这几条接缝就是"胶水";**定稿后再分给不同 session,否则拼不上**。

1. **控制面 ↔ 执行面(经 DB)**:UI 只写"定义 / 排程",并读"状态 / 产出";worker 只读"待执行的 Run",执行后写回"状态 / 产出 / Event"。两边都不直接调用对方。
2. **前端 ↔ 后端(REST / OpenAPI)**:FastAPI 自动生成 OpenAPI = 这份契约本身。概念上的资源面:`auth`、`workflows`、`agents`、`schedules`、`runs`、`outputs`、`events`。前端只依赖这份契约。
3. **后端/worker ↔ Agent 运行时**:worker 拿一个 WorkflowDef → 调 Agent SDK 执行 → 拿回结构化结果。这层接口要清晰(输入=定义+参数,输出=结构化产出+状态)。

## 5.5 模型 / Provider 抽象(多厂商支持)

目标:系统**既能用 Claude(Agent SDK),也能用别家模型**。靠一个抽象层解耦,别让编排代码绑死某一家(模型无关工程)。

**先按"要不要 agent harness"分,再按"哪家"分——不是单纯 Claude vs 非 Claude:**

- **简单调用**(总结/分类/抽取,给 prompt 要输出)→ 不用任何 agent SDK,**所有家(含 Claude)走一个统一裸调用层**(LiteLLM / OpenAI 兼容 / OpenRouter);简单任务路由到便宜或免费模型。
- **完整 agentic**(要工具、多步循环)→ 才分厂商:
  - Claude → **Claude Agent SDK**(吃它完整 harness + 内置工具 + 订阅/额度计费路径)
  - 非 Claude → **LangGraph** 或那家自己的 agent SDK

**一个架构岔路(取舍,自己定):**
- 方案 A:Claude 走自己 SDK、别家走 LangGraph → Claude 体验最好,但要维护两套 agentic。
- 方案 B:**全走 LangGraph(Claude 也走)** → 一套、干净、真正可换,但 Claude 失去专属优势。

**落地的缝**:`AgentDef` 声明 `provider/model`;执行层据此路由到对应实现。

**YAGNI:别现在就建多厂商系统。** 先**只用 Claude Agent SDK 单路径**跑通,把这条缝留好;等**真要接非 Claude 时**再加第二条路(简单调用加 LiteLLM,或 agentic 上 LangGraph)。设计缝,先实现一条。

## 6. 技术栈(含理由)

- **后端:FastAPI(Python)** — Agent SDK 的 Python 版是一等公民;异步适合 I/O 密集的 agent 活;自动产出 OpenAPI(=前后端契约)。
- **数据库:PostgreSQL** — 关系型可靠;JSON 字段存灵活的声明式定义;后期加 **pgvector** 做向量记忆(接上"长期记忆"学习目标)。
- **前端:React** — 控制面 UI(**后期阶段**才做);响应式覆盖电脑/平板/手机不同布局。
- **调度器:cron / APScheduler / 任务队列(Celery、RQ、Arq)** — 缺口里那块,触发定时工作流。
- **认证:别自己手搓** — 用成熟库/服务(fastapi-users、Authlib,或托管的 Auth0/Clerk/Supabase);密码 bcrypt/argon2 哈希;全程 HTTPS。
- **Agent 运行时:Claude Agent SDK(Python)** — 计费走 6/15 后那笔 Agent SDK 额度(练手规模够用)。
- **多厂商层(后期才接):LiteLLM**(统一裸调用,路由便宜/免费模型)/ **LangGraph**(provider 无关的 agentic)——见第 5.5 节,先留缝、用时再加。
- **托管:常开的云 VPS / 容器**(阶段 3)——这是"出门在外也能跑"的前提。

## 7. 安全与权限(横切质量带,单列强调)

- **认证**:登录是真实安全面(见第 6 节,用成熟方案)。
- **密钥管理**:API key、社媒凭据等**绝不写死在代码里**,用环境变量 / 密钥库。
- **Agent 权限**:用 Agent SDK 的权限模式,**逐 agent 最小权限**(每个 agent 能碰哪些工具/文件/数据,写进它的 AgentDef)。
- **meta-agent 护栏(到时候)**:一个能创建 agent、还能排程的 agent = 能让系统干任意事的 agent。必须配:强权限边界、**人在回路审批**它创建的东西、**审计日志(Event 表)**、沙箱隔离。

## 8. 增量路线图(建造顺序)

愿景当北极星,但一片一片建。每一阶段都是**能跑、能学到一个概念**的切片。

- **Phase 0 — 契约定稿**:本文档 + 数据模型 + API 资源面 + 定义格式敲定。**先于任何并行开工。**
- **Phase 1 — 后端单工作流端到端**:一个 session 顺序做通"抓新闻 → 解析 → 产出写进 DB"。验证接缝。
- **Phase 2 — DB 真相来源 + 调度器 + 通知**:6 点定时触发,产出推送给我(邮件/TG)。
- **Phase 3 — 控制面 web app**:认证 + 看状态/产出 + 排任务(React 先做最小)。
- **Phase 4 — 响应式 / 手机监控视图** + 更丰富的监控。
- **Phase 5 — 多 agent 编排**:subagent、orchestrator。
- **Phase 6 — meta-agent(带护栏)** —— 最后做。
- **(浏览器 / 社媒 agent**:按需插入,先过 ToS 合规。)

诀窍:**现在就按完整愿景把"缝"设计好(第 2、3、5 节),但只先实现第一片**——既不过度建设,又不堵死未来。

## 9. 怎么协调 build sessions(把多 agent 胶水落到建造上)

- **你是 orchestrator**,持有全局(这份文档)。别指望 session 自己组织。
- **契约先行,再开工**;greenfield 阶段**别并行撒一堆 agent**。
- **先用一个 session 顺序做通 Phase 1 的纵向切片**,拿到能跑的骨架 + 验证契约对。
- **然后才扇出**:一个 session 一个模块,各自对着本文档里定死的契约干。
- **Git 仓库 = 单一真相来源**(代码 + 本规约都在里面);用分支 / worktree 防止互相踩。
- **集成验证**:契约测试 + 集成测试 + CI 自动检查接缝。
- **你来 review 和整合**(orchestrator + verifier)。
- 给每个 session 的料:**本文档 + 它那块模块的具体契约 + 小而清晰的范围**(上下文才不乱)。
